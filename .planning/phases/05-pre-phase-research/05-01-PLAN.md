**第一章：Phase 前调研流程概览**
- 调研在 GSD discuss-phase 之前执行（per D-12: 互补关系）
  - 调研流程：产出 RESEARCH.md → GSD discuss-phase（基于调研结果讨论）→ GSD plan-phase（规划）→ GSD execute-phase（执行）
- 调研定位：为讨论提供信息基础，不替代讨论
- 调研范围：当前即将执行的 Phase 所需领域/技术背景

**第二章：轨道选择规则（per D-01 双轨调研制）**

写一个决策表：

| Phase 特征 | 选择轨道 | 说明 |
|-----------|---------|------|
| 涉及临床分析方法、文献引用、研究设计 | Track A（领域调研） | 搜索领域背景、最佳实践、文献 |
| 涉及 command、hook、workflow 改造 | Track B（技术调研） | 扫描 codebase + 搜索技术方案 |
| 同时涉及以上两者 | 双轨 | 先 Track A 再 Track B，优先综述性建议 |
| 纯配置、工具链升级 | Track B | 侧重技术对比选型 |
| 新统计方法、分析范式 | Track A | 侧重方法学文献和应用案例 |

判断方式：根据 Phase 名称和描述中的关键词自动匹配：
- Track A 关键词：分析、方法、文献、引用、统计、研究、临床、期刊
- Track B 关键词：命令、钩子、workflow、配置、工具、重构、脚本、文档

**第三章：调研深度自适应协议（per D-02）**

第一轮（摘要级发现）：
- 产出 3-5 个关键信息点
- 附带 2-3 篇参考来源（DOI/URL）
- 控制在"够做决策"的量级

用户追问时（深入层）：
- 展开原理说明
- 提供代码示例或实现细节
- 补充更多参考文献

明确规则：首轮输出必须控制在摘要级，禁止一次性输出全部调研内容。用户没有追问时不得自动展开。

**第四章：与现有 discuss 步骤的关系（per D-06, D-08）**
- 现有 discuss 步骤不变（D-08）
- 调研作为新增前置步骤执行（D-06）
- 调研产出 RESEARCH.md → 该文件内容自动流入 discuss 环节的 CONTEXT.md

在这个章节写明调研步骤的执行时机和触发条件：
- 调研步骤在 GSD `/gsd-plan-phase` 之前执行
- 当 planner agent 开始规划某个 Phase 时，先读取本参考文档，触发调研流程
- 调研完成后产出 RESEARCH.md，然后进入正常的 discuss → plan → execute 流程
- 不创建新 command（D-06）：通过现有 GSD 管理流程的状态检测触发

**第五章：调研结果流入 CONTEXT.md（per D-05）**
写明如何将 RESEARCH.md 的内容映射到 CONTEXT.md 的对应节：
- RESEARCH.md 的"关键发现" → CONTEXT.md 的 `<specifics>` 节

**第六章：Track A — 领域调研协议**

调研目标：搜索 Phase 相关的临床/方法学领域的背景知识、最佳实践、现有文献。

搜索策略：
1. 调用 reference-agent 的 `phase_research` 模式（Track A 子模式）
2. 搜索渠道：
   - 学术文献搜索：`python scripts/pubmed_search.py "{phase_keywords}" --max 5 --type review`
   - 补充搜索：`python scripts/tavily_search.py "{phase_description} clinical research best practice" --depth basic --max-results 5`
3. 搜索关键词构建规则：从 Phase 名称和 ROADMAP.md 的 Goal 描述提取核心关键词

产出格式：
- 调研主题与范围（一句话概括该 Phase 的调研范围）
- 关键发现列表（3-5 条，按重要性和置信度排序）
- 关键参考文献列表（2-3 篇，含 DOI/PMID）

注意事项：
- 参考文献必须有真实 DOI，禁止编造（继承 reference-agent 的核心规则）
- 如果搜索结果不足（<2 篇有效文献），标注"搜索结果有限，建议手动补充"
- 不修改 reference-agent 的输出格式，直接消费其搜索结果

**第七章：Track B — 技术调研协议**

调研目标：搜索 Phase 相关的工程技术方案、库选型、最佳实践。

搜索策略：
1. 读取 codebase 地图文件：
   - `.planning/codebase/ARCHITECTURE.md` — 了解当前架构
   - `.planning/codebase/CONVENTIONS.md` — 了解代码规范
   - `.planning/codebase/STRUCTURE.md` — 了解文件结构
   - `.planning/codebase/INTEGRATIONS.md` — 了解外部集成
   - 无需额外扫描逻辑（per D-10）
2. Tavily 搜索技术方案：
   ```
   python scripts/tavily_search.py "{technology_keyword} best practice implementation guide" --depth advanced --max-results 5
   ```
3. 如涉及库选型，搜索选项对比：
   ```
   python scripts/tavily_search.py "{library_a} vs {library_b} comparison" --depth advanced --max-results 3
   ```

产出格式（与 Track A 统一）：
- 调研主题与范围
- 关键发现列表（3-5 条）
- 可选方案对比表（如涉及选型）

**第八章：双轨模式**

当 Phase 同时需要领域调研和技术调研时：
1. 先执行 Track A（领域优先，确定临床方向）
2. 再执行 Track B（技术方案基于领域方向选择）
3. 输出合并为一个 RESEARCH.md：
   - 主题部分说明该 Phase 的双轨性质
   - 关键发现合并排序（领域发现在前，技术发现在后）
   - 对比表优先展示领域选项（如研究设计选择），再展示技术选项（如工具选型）
