# External Integrations

**Analysis Date:** 2026-05-05

## External Services

### NCBI E-Utilities API

- **Endpoints used:**
  - `https://eutils.ncbi.nlm.nih.gov/entrez/eutils/esearch.fcgi` тАФ Search PubMed and other NCBI databases
  - `https://eutils.ncbi.nlm.nih.gov/entrez/eutils/esummary.fcgi` тАФ Fetch record summaries
  - `https://eutils.ncbi.nlm.nih.gov/entrez/eutils/efetch.fcgi` тАФ Fetch full article XML
  - `https://eutils.ncbi.nlm.nih.gov/entrez/eutils/elink.fcgi` тАФ Link between databases
- **SDK/Client:** Custom Python client (`scripts/ncbi_client.py`) using the `requests` library with retry adapter (`urllib3.util.retry.Retry`).
- **Auth:** `NCBI_API_KEY` environment variable (optional). Without it: 3 req/s rate limit. With it: ~10 req/s (configured with safety margin at 9 req/s).
- **Databases queried:** pubmed, gene, protein, nucleotide, snp, clinvar, taxonomy, biosample, assembly, sra.
- **Rate limiting:** Enforced via `scripts/ncbi_client.py` `rate_limit()` function with module-level `LAST_REQUEST_TIME` global variable.
- **Consumers:** `scripts/ncbi_search.py` (multi-database), `scripts/pubmed_search.py` (PubMed-specific with MeSH expansion and natural language query parsing).

### Tavily AI Search API

- **Endpoint:** Tavily REST API (via `tavily-python` SDK)
- **SDK/Client:** `scripts/tavily_search.py` using `tavily.TavilyClient` from the `tavily-python` package.
- **Auth:** `TAVILY_API_KEY` environment variable (required). Key format: `tvly-...`
- **Rate limiting:** None documented in code тАФ relies on Tavily service-side limits.
- **Features used:** Search depth (basic/advanced), topic (general/news), up to 10 results, AI-generated answers, domain include/exclude filters, raw content retrieval, image search.
- **Consumers:** Reference Agent (`agents/reference-agent.md`) for supplementary literature search during Phase 3 writing.
- **Fallback:** If `TAVILY_API_KEY` is missing, Tavily search is unavailable; the system falls back to Claude's built-in knowledge for literature references (`docs/getting-started.md` `docs/getting-started.md` lines 371-375).

### Unpaywall API

- **Auth:** `UNPAYWALL_EMAIL` environment variable (optional, used as API identifier).
- **Usage:** DOI-based open-access status checking and PDF retrieval. Referenced in `agents/reference-agent.md` and `pipeline/workflows/writing.md`.
- **Consumer:** Reference Agent during Phase 3 full-text retrieval workflow: DOI -> Unpaywall -> pdf-reader.
- **Note:** No standalone Python script exists for Unpaywall тАФ retrieval is performed inline by the Reference Agent in Claude Code.

### GitHub

- **Repository:** `https://github.com/Side-Peng/clinpub.git` (from `package.json` `repository.url`).
- **CI/CD:** GitHub Actions:
  - `test.yml` тАФ Triggers on push to `main` and pull requests. Runs Python tests (pytest) and Node.js tests (`node --test`).
  - `release.yml` тАФ Triggers on `v*` tag push. Creates GitHub Release with auto-generated release notes via `softprops/action-gh-release@v2`.

- **Release artifacts:** `.tar.gz` archives and `checksums.txt` generated by CI (listed in `.gitignore`).

## Claude Code Ecosystem

### Skills (Built-in, referenced by clinpub)

| Skill | Purpose | Referenced By |
|-------|---------|---------------|
| **ncbi-search** | PubMed literature search | `CLAUDE.md`, `SKILL.md`, `agents/reference-agent.md` |
| **tavily** | Supplementary web search | `CLAUDE.md`, `SKILL.md`, `agents/reference-agent.md` |

These are Claude Code built-in skills installed via the Claude Code skill system. They are not part of the clinpub npm package but are called from within agent prompts.

### Hooks (Claude Code PreToolUse)

Three hooks are registered in `.claude/settings.json` during installation by `bin/install.js`:

| Hook File | Trigger | Integration Mechanism |
|-----------|---------|----------------------|
| `hooks/clinpub-workflow-guard.js` | Write/Edit | Reads `STATE.md` to check current phase. Blocks writes to directories belonging to future phases. Outputs JSON `{decision: "allow" | "block"}` over stdout/stderr. |
| `hooks/clinpub-phase-boundary.sh` | Bash | Reads `STATE.md` and milestone files. Checks prerequisite phase completion and data file existence before allowing analysis commands. |
| `hooks/clinpub-prompt-guard.js` | Read | Scans CSV/XLSX/TXT files for prompt injection patterns (25 regex patterns). Warns but does not block. |

Hook registration is **idempotent** тАФ `bin/install.js` removes existing clinpub hook entries before re-adding. All hooks have a timeout (3000-5000ms).

### Manifest Handoff System

Internal integration between agents via `MANIFEST.yaml` contract files (`pipeline/references/manifest-format.md`):

- **Analyst Agent** writes `MANIFEST.yaml` in `02_PreprocessedData/` (Phase 1) and `04_Outputs/` (Phase 2).
- **Reference Agent** writes `MANIFEST.yaml` in `Reference/` (Phase 3).
- **Writer Agent** writes `MANIFEST.yaml` in `05_Manuscript/` (Phase 3-4).
- **Consumer agents** validate required artifacts exist before processing.

This is a filesystem-only handoff contract тАФ no API calls, no message bus.

## Integration Risks

**Tavily API key requirement:** Tavily is the only service that is strictly required but lacks a meaningful free fallback. If `TAVILY_API_KEY` is not set, literature search degrades to NCBI-only (which is actually the primary source). The agent documentation says Tavily is used for "supplementary" search, so this is a moderate risk.

**NCBI API key not set:** The 3 req/s default rate limit is functional for interactive use but will be slow for batch reference mining. No hard failure, just a performance penalty. The `ncbi_client.py` provides helpful setup instructions when key is missing.

**Unpaywall without API key:** Uses email as identifier. Without `UNPAYWALL_EMAIL` set, PDF open-access lookup may not work reliably. Falls back to requesting user-provided PDFs.

**No MCP servers:** The codebase has zero MCP server configurations. All integrations are either (a) Python scripts making REST calls, or (b) Claude Code built-in skills. No `claude_desktop_config.json` or `mcp*.json` files found.

**No webhook endpoints:** The pipeline has no incoming or outgoing webhooks. It is a pull-style system тАФ all integrations are initiated from within Claude Code during a session.

**No caching layer:** PubMed search results are not cached between runs. Each session hits the NCBI API fresh. No local database or cache directory for API responses.
